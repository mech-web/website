Mechdown - a platform for writing rich documents

Mechdown is a platform for writing documentation and generating static and dynamic websites. It has a simple-to-write syntax based on Markdown, and can be used as a stand-in replacement. Mechdown comes with a super power though; since it's built off Mech, users gain access to a full computational engine inside of their document. This allows authors to bring their documents to life through incorporating code and computation directly into the document. Moreover, Mechdown brings literate programming to the Mech language, allowing programmers to embed rich documentation directly into their program source. This document is written in Mechdown, demonstrating the platform's features and capabilities. Currently Mechdown and Mech are in their infancy, so features are limited to static stite generation, but the goal is to eventually support dynamic sites, queryable documents, dashboards, and other dynamic applications in the near future.

In the last blog post, we had mentioned that Mech v0.2 should be more useful on its own without having to be a full-fledged language. It'll take time to get there, but we want it to be useful in the meantime so people will actually work in Mech. The analogy we made is that Mech v0.2 is like a car engine -- it's a useful self-contained thing which allows it to be modularly used in a wide range of vehicles for different purposes.

Today we are introducing the first of those vehicles for the Mech language: Mechdown, a platform for writing documentation and generating static websites, as well as serving them. It has a simple-to-write syntax based on Markdown, so it can be used for most things Markdown is used for. Mechdown comes with a suprise though -- since it's built off Mech, users gain access to a full computational engine inside of their document.

In this blog post we'll introduce Mechdown and the practice of "literate programming", talk about the history of this concept, and show how it can be practiced with Mech to create static websites.

Background: Literate Programming

According to Wikipedia [3], Literate Programming is:

a programming paradigm introduced in 1984 by Donald Knuth in which a computer program is given as an explanation of how it works in a natural language, such as English, interspersed (embedded) with snippets of macros and traditional source code, from which compilable source code can be generated.

We often think of source code as a language we use to communicate with machines, but really source code's primary purpose is to communicate with other humans. That's why we use natural language words for keywords, function names, variable names, comments and so forth -- because we need to be able to map programming concepts to familiar concepts in our brains, and that's usually done by drawing connections using familiar words and symbols.

For instance a function in source code that draws a circle might be called circle(). Or it might be represented by an actual circle symbol ○. But when the command to draw a circle meets the CPU, it will likely be represented as a series of machine instructions that, to a human, don't obviously resemble the notion of a "circle" at all!

Some languages are designed to be more human-readable than others. Python, which eschews curly braces in favor of semantic whitespace, is often praised for its readability. On the other hand Lisp with its deeply nested S-expressions, which are closer to a machine representation of source than a human one, is often criticized for being inscrutible.

Then there are array languages such as APL, J, and K, which are designed to work with arrays as their primary data structure. These languages allow for concise and expressive manipulation of arrays, often using a minimalistic syntax with no keywords, and expanded glyph alphabet. For example, in APL, you can sum all elements of an array with a single operator +/. Moreover, these operations are broadcast over arrays, so you typically don't code explicit loops.you can perform operations on entire arrays without needing explicit loops. While the powerful operators of these languages allow you to write very terse code, they are often critcized as being "write only" languages, as the lack of English signifiers in the code render them unreadable to people who are not versed in the language.

In this section we'll provide a comprehesive look at the history of literate programming, some notable literate programming systems, and some prior work that predates Mech in this area.

Markup Languages

Generalized Markup Language (GML)

GML (Generalized Markup Language) was developed at IBM in the late 1960s by Charles Goldfarb, Ed Mosher, and Ray Lorie. It introduced the idea of descriptive rather than procedural markup—authors marked sections of text according to their meaning (e.g., heading, paragraph), not their appearance.

GML tags were embedded in plain text and used macros to define document structure, typically processed by IBM's mainframe text formatting systems. It allowed users to separate content from presentation and laid the groundwork for SGML, XML, and ultimately HTML.

Example (GML-style tag syntax):

:h1.This is a Heading

:p.This is a paragraph in GML. The tag “:p.” marks the start.

As a follow-up to GML, the authors created a more formalized version called SGML (Standard Generalized Markup Language) in 1986. SGML was a meta-language—a system for defining other markup languages. It introduced formal rules for document structure using a Document Type Definition (DTD), and it directly led to the creation of HTML and XML.

Example (SGML with DTD):

<!DOCTYPE note [
  <!ELEMENT note (to, from, body)>
  <!ELEMENT to (#PCDATA)>
  <!ELEMENT from (#PCDATA)>
  <!ELEMENT body (#PCDATA)>
]>
<note>
  <to>Alice</to>
  <from>Bob</from>`
  <body>Meet me at 5pm.</body>
</note>

SGML is more verbose than GML, but it allows for more complex structures and relationships between elements. It was widely adopted in the 1990s for technical manuals, legal documents, and other structured content where precision and validation were critical.

TeX and LaTeX

TeX is a typesetting system developed by Donald Knuth in the late 1970s [5], designed for producing high-quality technical and mathematical documents. Unlike markup languages focused on structure, TeX is procedural—it gives the author fine-grained control over formatting and layout.

TeX uses commands prefixed with a backslash \ to control document behavior. It is especially useful for rendering complex mathematical notation, making it the standard in academic publishing -- even Mechdown uses TeX for rendering equations! 3 For example, this equation expressed in TeX:

\int_0^1 x^2 \, dx = \frac{1}{3}

is rendered as:

In 1985, Leslie Lamport created the LaTex system (stylized as ), which extended TeX to make it easier to produce books and articles [6]. His goal was to provide a more user-friendly interface for writing technical documents, allowing users to focus on content rather than formatting. For example:

\documentclass{article}
\begin{document}

\section*{Introduction}

This is a simple LaTeX document. Here's the same integral from before:

\[
\int_0^1 x^2 \, dx = \frac{1}{3}
\]

\end{document}

This system allows authors to write documents in separate files, and link them together using input or include. LaTeX also supports bibliographies, cross-references, and other features that make it suitable for large documents. It has become the de facto standard for typesetting academic papers, theses, and books in many fields, especially mathematics and computer science. One feature that supports this is stylesheets, which allow authors to define a set of formatting rules for their documents. This allows for consistent formatting across multiple documents, and makes it easy to change the appearance of a document without modifying the content.

Today, LaTeX is being developed and maintained by the LaTeX Project, which is volunteer-driven and open-source [7]. The LaTeX Project is responsible for maintaining the LaTeX kernel, which is the core of the LaTeX system, as well as a number of packages that extend its functionality.

HTML / CSS

HTML evolved from SGML in the early 1990s as a markup language for the web. It was develoepd in 1991 by Tim Berners-Lee, who wanted a way to share documents over the fledgling Internet while he was working at CERN. HTML is a simplified version of SGML, designed for easy authoring and rendering in web browsers. HTML defins a common set of tags, of the variety introduced in SGML, to describe common elements in web documents such as headings, paragraphs, links, and images.

Here's an example HTML document that uses the declaration from SGML to specify the document type, and presents a heading, paragraph, link, and image:

<!DOCTYPE html>
<html lang="en">
<head>
      <meta charset="UTF-8">
      <title>My Document</title>
</head>
<body>
      <h1>My Document</h1>
      <p>This is a paragraph in HTML.</p>
      <a href="https://example.com">Link to Example</a>
      <img src="image.jpg" alt="An example image">
</body>
</html>

Whereas HTML defines the structure and content of the document, CSS (Cascading Style Sheets) is used to control the presentation and layout. CSS allows authors to separate content from design, enabling consistent styling across multiple documents. Invent in 1996 by Håkon Wium Lie as a way to style HTML documents, CSS introduced a set of rules for applying styles to elements based on their tags, classes, and IDs. This separation of concerns allows for easier maintenance and updates to the visual appearance of web pages.

Like HTML, CSS documents are declarative in nature; they describe what the document should look like, rather than how to achieve that appearance. CSS rules are applied to HTML elements based on their selectors, which can be tags, classes, or IDs. HTML tags are assigned CSS classes or IDs, and CSS rules are applied to those elements based on their selectors. Here's an example of a CSS stylesheet that styles the tags used in the HTML document above:

h1 {
    color: blue;
    font-size: 24px;
}
p {
    font-family: Arial, sans-serif;
    line-height: 1.5;
}
a {
    text-decoration: none;
    color: #007BFF;
}

This clear separation of content and presentation is a common design pattern in document generation systems, and it allows for easy updates to the visual appearance of a document without modifying its content. This is especially important in web development, where the same HTML document may be viewed on a variety of devices with different screen sizes and resolutions, or to support accessiblity features like high contrast mode or screen readers.

Javadoc et al.

As codebases grew larger in the 1990s, developers needed better ways to document software directly alongside source code. This led to the rise of inline documentation systems—special comment formats that could be automatically extracted to produce human-readable documentation.

Java pioneered this approach with Javadoc, introduced by Sun Microsystems with the original Java Development Kit (JDK). Javadoc uses comment blocks marked by /** */ and tagged fields like @param, @return, and @throws to describe methods and classes:

/**
 * Returns the maximum of two integers.
 * @param a the first number
 * @param b the second number
 * @return the greater of a and b
 */
public int max(int a, int b) {
    return (a > b) ? a : b;
}

Using the javadoc tool, developers can generate consistent HTML documentation automatically from their source files, promoting better code readability and standardized external docs.

Other languages quickly adopted similar systems. Doxygen supports C, C++, and several other languages with a syntax similar to Javadoc. Rustdoc generates documentation from triple-slash /// comments in Rust:

/// Adds two numbers together.
/// 
/// # Arguments
/// 
/// * `a` - The first number.
/// * `b` - The second number.
/// 
/// # Returns
/// 
/// The sum of `a` and `b`.
fn add(a: i32, b: i32) -> i32 {
    a + b
}

Rustdoc imporves upon Javadoc in several ways:

  • Markdown support: Documentation comments are parsed as full Markdown, allowing easy formatting (e.g., headings, lists, code blocks, links).

  • Executable code examples: Code snippets written in the documentation are automatically compiled and tested with cargo test. This ensures examples are always correct and up to date.

  • Hidden setup code: Rust allows parts of code examples to be hidden from the rendered documentation (using # to hide lines), making examples clean for the reader while still functional during testing. For example:W

These additional features try to solve the problem of keeping documentation synchronized with code, which is a common issue in large codebases. By allowing documentation to be written in Markdown and including executable code examples, Rustdoc makes it easier for developers to maintain accurate and up-to-date documentation.

Markdown and Extensions

Markdown was created by John Gruber in 2004 with the goal of making documents easy to write and easy to read in plain text. Its syntax is intentionally simple: headings are marked with #, lists with -, links with text, and code blocks with backticks. Unlike earlier markup languages, Markdown prioritizes readability first, even without any processing, so that plain-text documents would appear as close to their rendered form as possible.

For example, this Markdown document:

# This is a Heading

## This is a Subheading

This is a paragraph with a [link](https://example.com).

- List item 1
- List item 2
- List item 3

~~~
This is a code block.
~~~

Because it's easy to read and write and implement, Markdown quickly gained popularity and became the de facto standard for lightweight markup languages. Along the way it gained several notable extensions like GitHub Flavored Markdown (GFM), which added support for tables, strikethrough, and task lists; and a spec called CommonMark, which aimed to standardize the syntax across implementations.

History of Literate Programming

Literate programming is a programming paradigm introduced by Donald Knuth in 1984, when he introduced the WEB systems as a way to improve the readability and maintainability of code1. Knuth was especially frustrated with the limitations of traditional programming languages, which required programmers to write code in a compiler-imposed order that didn't make sense for the narrative of explanatory works like books and academic papers. The goal was to write code in a way that was more natural and readable to explain his ideas more clearly. His insight was that programs should be written primarily for human understanding, not just machine execution.

In his words:

The practitioner of literate programming can be regarded as an essayist, whose main concern is with exposition and excellence of style. Such an author ... strives for a program that is comprehensible because its concepts have been introduced in an order that is best for human understanding, using a mixture of formal and informal methods that reinforce each other [8].

For example, at the time some languages required programmers to declare all variables before you any of them were used, like early Pascal, Fortran, and Algol. This was only the case because it made the lives of Pascal compiler developers easier, but the purpose of a language should be to make the users' lives better, not the lang devs.

As a programmer, you would rather declare a variable where it makes narrative sense, so that the reader can see how it is used in context rather than be refferred to a distant section of the text. It shouldn't matter how the compiler feels about it.

Some of this can (and has) been solved with better language design; modern languages are much more free about where they allow you to declare variables. But the underlying point still remains -- the compiler shouldn't have any say in the order we write our program in just because it might be more conformant to the machine's representation of the program.

While Knuth's tooling was cutting-edge in 1984, his work only just began to reveal the potential of this direction. Let's take a look at some of the different implmentations of literate programming systems over the years, and how modern literate programming systems are used today.

Notable Literate Programming Systems

WEB

WEB was the the foundational tool for literate programming developed by Knuth in the 80s. It was designed to interleave Pascal code with TeX-formatted documentation2.

WEB consists of three major components:

  • Source File (.web) - This is the document written by the programmer. It contains both documentation and code, interwoven in sections. Each section is like a small story fragment that may include prose followed by code, which can reference or expand other sections by a name reference.

  • TANGLE - This tool processes the .web file and extracts the code portions, assembling them into proper Pascal source code. The name reflects the idea of untangling the narrative to produce compilable code. It resolves named code sections into an order appropriate for compilation, and outputs a .pas file, which can be compiled by a Pascal compiler.

  • WEAVE - This tool turns the .web file into a typeset document using TeX. It organizes and formats both the prose and the code with syntax highlighting, cross-references, and indexing, generating high-quality documentation -- typically a .tex file for further processing into PDF.

So the workflow went like this:

filename.web ──► TANGLE ──► filename.pas ──► [Pascal Compiler]
             └──► WEAVE  ──► filename.tex ──► [TeX Typesetting]

Literate Haskell

Haskell is one of the few languages that provides native features to support literate programming [9]. In Haskell, a literate program is one with the suffix .lhs rather than .hs.

This literate program prompts the user for a number
and prints the factorial of that number:

> main :: IO ()

> main = do putStr "Enter a number: "
>           l <- readLine
>           putStr "n!= "
>           print (fact (read l))
          
This is the factorial function.

> fact :: Integer -> Integer
> fact 0 = 1
> fact n = n * fact (n-1)

It's kind of an inversion of the typical code/comment syntax; whereas as in typical languages you normally write code and use special sigils to denote comments, in literate haskell you normally write comments and use special sigils to denote code. Obviously this will become noisy if you have a lot of code to write, so Haskell also allows you surround lines of code with begin{code} and end{code}. This is a little more verbose, but it allows you to write code in a more natural way.

When fed into a Haskell compiler like GHC, only the lines prefixed with > will be compiled, while the rest are treated like comments. When fed into a documentation generator though all lines are parsed into rich documents in the target format.

Here's another example wher LaTex is used to format the prose with a section header and a code block.

\documentstyle{article}

\begin{document}

\section{Introduction}

This is a trivial program that prints the first 20 factorials.

\begin{code}
main :: IO ()
main =  print [ (n, product [1..n]) | n <- [1..20]]
\end{code}

\end{document}

Emacs Org-mode

Whereas Literate Haskell is specific to the Haskell programming language, Emacs Org-mode [10] is a general-purpose markup language that can be used with any programming language. It was developed by Carsten Dominik, Bastien Guerry et al in 2003 as a way to organize notes, tasks, and projects in Emacs. Org-mode allows you to write documents in a plain text format, and then export them to various formats such as HTML, LaTeX, and PDF.

Org-mode is so called because it's designed to help organizations use Emacs by giving it features that are useful for that kind of thing. Support for literate programming is an obvious win here, because what better way to help a team stay on top of a project than support for rich documentation?

Notebook Programming Systems

Notebook programming is the most recent incarnation of literate programming, and possibly the most useful to date. Notebooks are are very close to what Mechdown is, but Mechdown improves on them in important ways. In this section we'll take a look at two notable notebook programming systems: JuPyter and Matlab.

Jupyter

Notebook Programming is a language-agnostic concept, but Python and Julia have particularly been used for notebook programming due to their prevalence in scientic and engineering fields, and the JuPyter project is a notebook for precisely these languages. The way it works is that the user writes prose in Markdown, and then writes Python or Julia code blocks behind a markdown code fence.

Key Features of Jupyter Notebooks:

  • Markdown Cells: You can use Markdown to write explanatory text alongside your code.

  • Code Cells: Code is written in cells that can be executed independently.

  • Interactive Output: The output of code, whether it’s data, graphs, or text, is displayed directly below the code block.

  • Reproducibility: Code and results can be saved together in the notebook, ensuring that others can reproduce the work.

  • Support for Multiple Languages: While Jupyter originally supported Python, it now supports dozens of languages, including Julia, R, and even JavaScript.

Matlab

Matlab is Mech's closest cousin and also a language that has a long history of being used for notebook programming, especially in engineering, physics, and numerical analysis. Matlab's primary strength lies in its powerful array and matrix operations, making it an ideal tool for mathematical modeling, simulation, and data analysis.

Matlab has supported a notebook-like interface since version 7, where users could combine script execution with rich text annotations. This style of interactive programming allows users to create self-contained documents that explain and execute code in a cohesive manner.

  • Live Scripts: Matlab's live scripts allow for combining code and formatted text in a single document. These scripts are similar to Jupyter notebooks in that they allow inline results and visualization.

  • Interactive Plots: Matlab supports interactive plots that can be adjusted within the notebook itself, allowing for dynamic exploration of data.

  • Rich Outputs: Matlab supports rich outputs like equations, formatted text, and interactive visualizations directly in the script.

  • Built-In Functions: Matlab comes with a vast library of built-in functions for numerical analysis, signal processing, optimization, and more.

Mechdown Architecture

Mechdown and Mech are tightly integrated, to the point that Mechdown is implemented in the Mech parser. Writing Mech code in the Mechdown dialect is idiomatic, but of course you can write Mech programs without writing any Mechdown at all.

Parsing

  • Mech programs are defined as a title with sections, each section has a optional subtitle and a body, the body can consist of section elements and that includes all the things we can render, as well as mech code.

  • Mech code is partially whitespace sensitive, so we don't strip out all white space intially.

  • This allows us to be very flexible about how we write code, because we don't have to rely on braces and brackets and semicolons to deliniate sections since we didn't blow away all the whitespace.

  • But it's not like python in that you indent to define blocks, because that gets problematic, so really everything is separated at the newline level.

  • What does this buy us? A we can write naturally just as in markdown.

  • We can then write code naturally, just as in a regular programming language. If you don't write any code, writing mech can feel a lot like writing matlab at this point.

  • Parsing generates a parse tree which we then pass on to the formatter

Formatting

  • The formatter takes a parse tree and generates a HTML document from it.

  • The HTML document is fully styled

  • The formatter works by walking the tree and building an HTML tree, so really the HTML document can be thought of as another representation of the parse tree. This is neat becaues we can use Chrome's element inspector to inspect the parse tree, and we can use the Mechdown formatter to generate a new parse tree from the HTML document. Well, we could anyway, but what we actually do is we store a BASE64 encoded version of the parse tree from which the document was generated inside of the document, so that we don't have to reparse anything if we want to recover the source code.

  • The formatter can also include a CSS document to custom style the HTML. This makes it flexible and we can apply themes to alter the appearance or structure of the document. We include enough nodes in there that it should be possible to get a wide array of styles.

  • The formatter can take a single .mec file or it can take a directory or multple files, and it will format an output directory of HTML files. IT can accept .mec files for now, but in the future it could accept .md and .html files and convert them.

  • You can upload the result to a static webpage server, or you can include the Mech tool as a step in your CI build process, so that you just upload your .mec file, and then use mech format as to automatically generate the HTML documents.

Serving

  • mech format gets you a nice document, but it's static until it's connected to a Mech runtime. What's cool about Mech is it compiles to web assembly, so we can fit the entire runtime in a 1.3MB (compressed) .wasm file that we can ship to a client browser, so the program can be executed right there on their machine. This is a big win over systems that need a server to execute client code.

  • So how does this work, the html document loads the wasm file, which then starts a mech interpreter, which then gets the Base64 encoded tree, which then executes it and we get the result for every variable in the system.

  • After init, Mech goes through the document to any values that need to be reified, and we replace the expression with the result.
